**Dataset:** cluster distribution, n=65536 keys, 5 Gaussian clusters, 15% uniform noise, 60-bit keys, 200k queries (skip-mode FPR).

## Grafite: корректная работа на кластерах

Grafite — **hash-based** фильтр. Строит Elias-Fano по схеме:

```
h(x) = ( (a*(x/r) + b) % p  +  x ) % r
```

Хэш-функция равномерно перемешивает ключи по compressed universe `r`, независимо от их оригинального распределения. После хэширования кластерные ключи неотличимы от равномерных — Elias-Fano работает оптимально. Результат: FPR снижается с ростом BPK так же, как на uniform данных, и достигает 0 при BPK ≥ 20.

## SNARF: структурный floor FPR ≈ 0.707

### Как работает SNARF

1. Обучает **RMI (Recursive Model Index)** на отсортированных ключах — аппроксимирует CDF
2. Создаёт bitset размером `n × 2^bpk` бит; для каждого ключа устанавливает бит на позиции `predicted_rank × 2^(bpk − log₂n)`
3. Запрос `[lo, hi]`: предсказывает `rank(lo)` и `rank(hi)`, проверяет наличие бита в этом диапазоне

### Почему FPR не снижается с ростом BPK

FPR имеет два слагаемых:

```
FPR_total = FPR_model + FPR_bitset
          = P(ε_model > block_size) + P(hash collision | ε_model ≤ block_size)
```

- **`FPR_bitset`** — коллизии внутри блока — экспоненциально убывает с BPK
- **`FPR_model`** — ошибка RMI предсказания превышает размер блока — не зависит от BPK

На **кластерном датасете** RMI обучается на 5 плотных кластерах. В разреженных межкластерных зазорах модель не видела примеров при обучении. Для пустого запроса `[lo, hi]`, попавшего в такой зазор:

```
predicted_rank(lo) ≠ true_rank(lo)  →  смотрим не туда в bitset  →  false positive
```

Ошибка модели в sparse регионах систематически велика (модель «прыгает» через зазор, отображая его в ближайший кластер). Когда `ε_model >> block_size`, дополнительные биты не помогают — `FPR_total → FPR_model ≈ const`.

### Почему именно ~0.707

Это доля истинно-пустых запросов, для которых ошибка RMI необратимо велика. Определяется геометрией кластеров: 85% ключей в 5 кластерах создают ~85% "опасных" зазоров, и ~70% пустых запросов в skip-mode попадают именно туда. Число устойчиво к BPK — подтверждает, что floor задаётся не bitset-ом, а моделью.

### Контраргумент (SNARF на uniform)

На равномерном датасете RMI аппроксимирует линейную CDF с малой ошибкой → `FPR_model ≈ 0` → FPR контролируется только BPK. Это объясняет, почему оригинальная paper SNARF (тест на YCSB/OSM/uniform) показывала хорошие результаты.

## SuRF

SuRF — **trie-based** фильтр, хранит compressed FST (Finite State Transducer) по ключам.

- **SuRF(None)** и **SuRFHash(8)** дают одинаковый FPR 0.165 при L=1 — хэш-суффикс помогает для point queries, но не для range queries, потому что trie-ветки ограничены по разрешению
- **SuRFReal(8)** даёт FPR 0.0005 при 41.9 BPK — real suffix bits хранят реальные биты ключа, что резко снижает FPR range-запросов
- SuRF не параметризован BPK напрямую: размер зависит от trie-структуры (ветвление, длина ключей)
- Аномальный размер SuRFHash/SuRFReal (41.9 BPK) — ожидаем: FST на 60-битных uint64-ключах хранит 8-байтовые строки + суффиксные bits

## Итог для тезиса

| Подход | Distribution-dependent | FPR на кластерах | Масштабируется с BPK |
|--------|----------------------|-----------------|---------------------|
| Grafite (hash) | нет | да, хорошо | да, экспоненциально |
| SNARF (learned) | **да** | **нет**, floor ~0.7 | нет (floor = model error) |
| SuRF (trie) | нет | посредственно | нет (фиксированная структура) |
| SuRFReal (trie+real suffix) | нет | хорошо при большом BPK | частично |
| ARE (Adaptive/SODA) | нет | да, оптимально | да, BPK = log₂(L/ε) |

**Вывод:** SNARF неприменим для кластерных рабочих нагрузок из-за model error floor. Grafite работает хорошо, но требует ~2 BPK overhead по сравнению с ARE-нижней границей. ARE-фильтры (Adaptive, SODA) достигают теоретической нижней границы `log₂(L/ε)` на кластерных данных.
